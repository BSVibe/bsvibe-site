BSNexus is built on a small vocabulary and four surfaces. Internal architecture (execution runs, composition snapshots, tool calls, workers) lives behind the **Inside** surface — visible when you ask for it, invisible the rest of the time.

## Vocabulary

| Term | What it is |
|------|------------|
| **Direction** | Your instruction. A short natural-language input that tells BSNexus what to do or how to change existing work. |
| **Request** | The user-facing unit of work derived from a direction. Holds intent, status, related decisions, related deliverables, and the current next step. |
| **Decision** | A judgment AI should not make alone — taste, scope, risk, budget, deploy approval, destructive action, external communication. |
| **Brief** | A skimmable status summary: what shipped, what needs your decision, what is blocked, what is running, and what BSNexus recommends next. |
| **Deliverable** | A reviewable output of work — code, PR, preview URL, design, document, marketing copy, report. |
| **Proof** | Typed evidence attached to a deliverable: tests, exit codes, screenshots, PR diffs, source links. Logs are not proof. LLM confidence is not proof. |
| **Inside** | The advanced inspection surface where execution runs, composition snapshots, and tool calls live. Optional, never the default view. |

## The Four Surfaces

A project has four surfaces, all built on the same primitives:

### Direction

A single conversation per project. You write what you want; BSNexus classifies the message (chit-chat, question, request, modification) and creates or updates a Request accordingly.

You do not pick agents, fill in templates, or split work yourself. The Direction surface is the only chat in BSNexus.

### Progress

A timeline of work and three trust cards showing whether your knowledge layer (BSage), gateway (BSGateway), and audit layer (BSupervisor) are connected. As BSNexus matures, this surface becomes a Brief — shipped, needs decision, blocked, running, next.

### Decisions

The approval inbox, blocking items first. Each Decision card includes the question, the recommendation, the options, the impact, the affected request, and a resume note. You resolve a Decision in one tap and the blocked work continues.

### Inside

Optional. The execution-run tree, the composition snapshot for each run, and the tool-call audit. Use it when you want to debug or learn what BSNexus did. The default workflow does not require opening it.

## Proof before done

BSNexus says "done" only when it can show proof.

For software work, the minimum proof is:

- the files changed or PR link
- the verification command that was run
- the exit code and a short summary
- known risks or limitations

Preferred proof adds tests, typecheck/lint, preview URL, screenshot for UI, and CI status. Cheerful summaries are not proof. A deliverable with `verification_failed` or `verification_missing` will not be presented as shipped.

This applies to design, documents, and marketing too. Every deliverable type has a proof model — rendered preview and rationale for design, source links and validation for documents, target audience and channel checklist for marketing copy.

## What BSNexus is not

- It is not a prompt-to-app builder. The work continues after you leave, and the output is meant to be maintained.
- It is not a board or task management tool. Direction, Decisions, and proof are the surfaces — there is nothing to drag around.
- It is not an agent or org-chart dashboard. You do not configure workers, personas, or routing in the default workflow.
- It is not a chat UI that summarizes LLM output. Done means proof.

## Interfaces

BSNexus is interface-agnostic. Web is one client. Mobile web/PWA is the first computer-independent interface, so you can resolve decisions and review deliverables from your phone. Slack, email, CLI, and voice follow later, and all map to the same primitives: Direction, Decision, Brief, Deliverable, Proof.
